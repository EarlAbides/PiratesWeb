## What Went Well

1. **Zero technical debt carried forward.** 25+ review findings across 4 stories — every single one resolved in-story. The discipline established in Epic 1 held perfectly.

2. **Class-based store pattern scaled cleanly.** `filterState.svelte.ts` started as sort-only in 2.2, extended with 5 filter properties + derived count + `clearAllFilters()` in 2.3, then had `matchesSearch()` extracted in 2.4 — zero rework at each step.

3. **Sandbox as parallel design proving ground.** The ship card layered design, PNG stat icons, cannon pips, and set backgrounds all landed via sandbox sessions running outside the story cycle. Story Dev Notes simply said "do not touch" and the implementation stayed clean.

4. **Visual identity elevated the whole epic.** Captain's words: the sandbox work "really elevated the whole epic." The Cinzel typography, PNG stat icons, and layered card design produce something that reads as a game artifact, not a web app.

5. **filterUtils extraction made Story 2.4 effortless.** Story 2.3 proactively extracted filter/search logic to a testable pure module. Story 2.4 dropped cleanly on top — extracted one function, added 8 tests, done.

6. **Test suite grew consistently.** 12 → 17 → 28 → 36 tests. Each story built on the suite without regressions.

## Challenges

1. **Cannon pip CRITICAL bug (Story 2.2).** `parseCannonPip` expected type-first format ("S3"); `cards.json` uses digit-first ("3S"). Zero cannon pips rendered on every ship card. Unit tests passed because fixtures used the wrong format. Caught in code review — not during implementation. A real user-facing defect that shipped only to review, not to production.

2. **SVG anchor bias in sandbox sessions.** The team defaulted to SVGs for stat icons because they're the "correct" answer for web icons. The PNG alternative was faster to implement and visually superior (hand-drawn quality reads as a game artifact). Time was spent proving the SVG path before the assumption was challenged.

---

## Key Insights

1. **Synthetic test fixtures aren't enough for parsers that consume real data formats.** Any function that parses `cards.json` data needs at least one real data sample in the test fixture alongside synthetic ones. Format assumptions are invisible until they hit actual data.

2. **Challenge default technical assumptions on day one.** In sandbox sessions, put competing approaches side by side from the start rather than defaulting. The SVG detour cost design time that PNGs made irrelevant.

3. **The sandbox pattern is worth formalizing.** For visual-heavy epics, running a parallel sandbox sprint before stories begin — design sessions, Captain approves section by section, then implementation follows — is a first-class workflow, not an ad-hoc detour.

4. **Code review is load-bearing.** The cannon pip CRITICAL was invisible during implementation and testing. It only surfaced in review. Review must remain a mandatory step.
